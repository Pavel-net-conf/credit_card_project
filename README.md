# Сервис прогнозирования дефолта по кредитным картам с контейнеризацией и A/B-тестированием

## 📌 Описание проекта
**Цель проекта:** Разработка и подготовка к интеграции в банковскую production-like-среду веб-сервиса машинного обучения для прогнозирования дефолта по кредитным картам. Проект охватывает полный цикл: от разведочного анализа данных и обучения моделей до создания масштабируемого веб-интерфейса (API) с поддержкой одновременного версионирования для A/B-тестирования.

- **Домен:** Финансы / Кредитный скоринг.
- **Датасет:** Default of Credit Card Clients Dataset (UCI Machine Learning Repository).
- **Целевая переменная:** `default.payment.next.month` (1 — дефолт в следующем месяце, 0 — своевременная оплата).

 ### Сервисная архитектура и MLOps концепты

 1.Монолитный подход: Для данного веб-сервиса выбран монолитный подход, так как система на текущем этапе решает одну изолированную бизнес-задачу (скоринг). Микросервисная архитектура создала бы избыточную сетевую нагрузку (latency) и усложнила бы поддержку инфраструктуры без реальной бизнес-необходимости.
 
 2.Брокеры сообщений (RabbitMQ): При росте нагрузок синхронные HTTP-запросы заменяются асинхронными. Внешняя банковская система отправляет пакет заявок в очередь RabbitMQ, а наш сервис (в роли Consumer) постепенно разбирает их батчами. Это полностью исключает падение API по таймауту и сглаживает пиковые нагрузки.

 3.Логирование и мониторинг: API логирует данные инференса в структурированном формате JSON. В промышленной среде эти логи собираются агентом Filebeat, отправляются в связку Logstash / Elasticsearch и визуализируются в Kibana (ELK-стек) для отслеживания деградации моделей (Data/Model Drift) и графиков распределения предсказаний.
 
 4.uWSGI + NGINX: В production-среде встроенный сервер Flask использовать нельзя, так как он однопоточный. NGINX ставится на передовую как Reverse Proxy (буферизация запросов, защита, SSL, статика), а uWSGI/Gunicorn эффективно управляет процессами Python, распределяя входящие запросы по ядрам процессора.
 
 5.Преобразование в формат ONNX-ML: В целях оптимизации и снижения задержек (latency) инференса в production-среде scikit-learn пайплайны могут быть конвертированы в формат ONNX (с помощью библиотеки skl2onnx). Это позволяет выполнять вычисления на высокооптимизированном C++ рантайме (onnxruntime), снижая потребление RAM и ускоряя инференс в несколько раз.
 
 6.DVC и MLflow: DVC используется для контроля версий весов моделей .joblib и тяжелых исходных датасетов без захламления репозитория Git. MLflow применяется для трекинга параметров экспериментов, логирования метрик обучения и прозрачного сравнения v1 и v2 моделей.
 
 7.Бизнес-метрики: Помимо технических метрик ($F_1$, Accuracy), модель напрямую оптимизирует финансовую метрику Expected Loss (Ожидаемые финансовые потери) банка:

 
 $$EL = P(Default) \times EAD \times LGD$$

 
Где $P(Default)$ — выдаваемая моделью вероятность дефолта, $EAD$ — сумма долга. Сервис позволяет превентивно отказывать или снижать лимиты высокорисковым клиентам, сохраняя оборотный капитал банка. Вторая метрика — Approval Rate at Fixed Risk — увеличение доли одобренных заявок надежным клиентам при удержании общего уровня риска портфеля.

### Организация A/B-тестирования


1. Постановка эксперимента
   - Контрольная группа (A): Текущая рабочая модель (v1 — Логистическая регрессия). Получает 50% входящего трафика скоринга.
   
   - Тестовая группа (B): Новая улучшенная модель (v2 — RandomForestClassifier, учитывающий дисбаланс и важность признаков, таких как история платежей PAY_0). Получает 50% трафика.
   
   - Способ разделения трафика (Сплитование): Осуществляется на основе детерминированного хэширования идентификатора пользователя: Hash(User_ID) % 100 < 50 $\to$ группа A, иначе группа B. Это гарантирует, что при повторном запросе в течение эксперимента клиент всегда будет попадать на одну и ту же модель.
   
   - Продолжительность теста: 14 дней (минимум два полных финансовых недельных цикла для исключения влияния дней недели и сезонности выплат).

3. Метрики и статистический анализОсновная метрика:
   - $F_1$ -score для класса дефолта (1).
     
   - Дополнительная метрика: Recall (Полнота) для класса дефолта. Обоснование выбора: в кредитном скоринге пропустить дефолтного клиента (ошибка II рода) обходится банку в десятки раз дороже, чем ложно отказать хорошему клиенту (ошибка I рода). Нам критически важно максимизировать Recall, чтобы обнаружить как можно больше будущих банкротов.
   
   - Статистический метод: Двувыборочный Z-критерий для долей (Z-test) с фиксированным уровнем значимости $\alpha = 0.05$. Доверительные интервалы для метрик будут строиться непараметрическим методом Bootstrap (бутстрэп) с помощью повторных выборок.
   
   - Критерий успешности эксперимента: Получение статистически значимого роста метрики Recall в тестовой группе B ($p\text{-value} < 0.05$) при условии, что общая доля одобрений (Approval Rate) не упала ниже критического бизнес-порога.
